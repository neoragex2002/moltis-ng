# Telegram 群聊里「@ 转写 / 改写」机制（现状梳理 + 举例）

> 范围：**只描述当前代码已实现的行为（as-is）**，不讨论 V4 未来机制（WAIT/RootMap/TaskCard/epoch 等）。
> 目标：把群聊里跟 `@xxx` 相关的“改写/转写/镜像/中继”场景讲清楚，避免误读。
>
> 更新（2026-03-08）：本仓库已实现可选的 **TG-GST v1** 群聊 session transcript 文本协议（更自然的 `<speaker><addr_flag>: <body>` 形式；保留换行与 `@mentions`；在启用时移除 `[@... mirror]` / `（来自 ...）` 等 legacy 前缀）。详见：`issues/issue-telegram-group-session-transcript-text-protocol-tg-gst-v1.md`。

## 0. 阅读指南：先分清“系统机制” vs “会话文本协议”

这份文档里混着两类东西，不先分清很容易吵不明白：

- **系统机制**：系统“做不做一件事、把消息送给谁、写到哪里、是否触发推理、是否挂 reply 线程、如何避免重复/循环”等。
  - 例子：群里没点名时不触发 LLM、但要不要旁听入库；bot A 出站后要不要触发 bot B；bot B 的回复要不要挂到 bot A 那条消息下面。
- **会话文本协议**：写进 session 的那条“纯文本 content”长什么样（因为模型主要看到的就是这个文本，而不是系统元数据）。
  - 例子：是否要把 `@我` 从正文里删掉；mirror/relay 写进会话时用什么前缀；是否在文本里标注“谁说的/是不是叫我”。

本文件的结构是：
- **0（术语）**：先统一口径（建议先扫一眼）
- **0.5**：系统机制的人话版总览（先把流水线讲清楚）
- **1～7**：把各个场景拆开讲（每个场景会标注“偏机制/偏协议/两者都有”）
- **8**：已知问题（哪些地方容易误读、容易断链、容易重复/循环）
- **8.5**：当前讨论中你提出的“建议处理方式/取向”（先记口径，后续再细化）
- **9**：对应代码位置

建议阅读顺序：先看 **0（术语）**，再看 **0.5（系统机制）**，最后按 **2～7（场景）** 查细节。

## 0. 术语（口径先统一）

- **bot self-mention stripping（自我点名清理）**：在某个 bot 侧，收到的入站文本里，把 `@本 bot` 这段**删掉**，再做空白归一化后入库/送 LLM。
- **listen-only / ingest-only（旁听入库）**：群聊里**未点名**某个 bot 时，该 bot **不触发 LLM、不回复**，但把消息写入该 bot 对应的群聊 session 历史。
- **mirror（镜像）**：bot A 在群里发出的“出站回复”，会被写入 bot B/C 的群聊 session 历史，作为“旁观记录”。
- **relay（中继/转派）**：bot A 在群里发出的“出站回复”里，如果包含对 bot B/C 的指令式 `@botB ...`，网关会把这段**转成** bot B/C 的“入站触发”，让 B/C 被动开始干活，并在群里回复到 A 的那条消息下。

## 0.5 基础机制（人话版：系统到底在干什么）

> 这段只讲“系统机制/流水线”，不讨论最终应该把消息改写成什么文本格式（那属于“会话协议”）。

### 0.5.1 每个 bot 都有一份“这个群的记忆本”

- 同一个 Telegram 群里，每个 bot 都维护一份自己的会话历史（可以理解为：这个 bot 专属的“群聊聊天记录/上下文”）。
- 这份历史里会写入：群里的人说的话、这个 bot 自己在群里发出去的话、以及系统为它补偿写入的“别的 bot 的相关内容”（见后文 mirror/relay）。

### 0.5.2 群里来一条消息，系统先做分流：让不让这个 bot 运行？

对每个 bot，系统会先判断“这条消息要不要让它动脑/要不要让它回群”：

- **没点名**（且该 bot 配置为“只有点名才工作”）：
  - 不触发 LLM、不回群；
  - 但会把消息写进该 bot 的会话历史（让它“知道群里发生过这句”，只是暂时不参与）。
- **点名了它**（或该 bot 配置允许不点名也工作）：
  - 会把消息写进该 bot 的会话历史；
  - 然后触发一次推理，并把结果回到群里（通常用 reply 挂到那条触发消息下面）。

### 0.5.3 Telegram 的平台限制：bot 之间在群里“经常听不见”

- 在 Telegram 上，bot A 在群里发的消息，并不会像“人发的消息”那样，被可靠地投递成 update 给 bot B/C。
- 结果是：如果系统不做补偿，多个 bot 在同一个群里会互相“看不见彼此说了什么”，协作会断链。

### 0.5.4 系统的两类补偿动作：让 bot 之间“能看见/能交接”

当某个 bot 在群里成功发出回复后，系统会在后台做两类补偿：

- **转述（mirror）**：把 bot A 刚刚发出去的那段回复内容，作为“旁观记录”，写进其他 bot（B/C/...）各自的会话历史里，让它们能“看见 A 说了什么”。
  - 重要限制：只写进“这个 bot 在这个群里已经存在的会话”，避免给根本不在群里的 bot 凭空造出一份“假群会话”。
- **转派（relay）**：如果 bot A 的回复文本里明确出现“叫 bot B 干活”的写法（例如行首 `@botB ...` 这种强信号），系统会直接触发 bot B 的一次推理，把任务交给它。
  - 同时，系统会把 bot B 的回复目标绑定到 bot A 刚刚那条“真实发出去的消息”上，所以 bot B 的回复会挂在 bot A 那条消息下面（线程自然、能追溯）。

### 0.5.5 防止乱套的护栏（别误触发、别无限循环）

- **去重**：同一条转述/转派不会反复触发写入或触发（避免重复）。
- **不凭空拉人入群**：目标 bot 没有这个群的既有会话，就不转述/不转派（避免“幻觉成员/幽灵会话”）。
- **转派跳数上限**：允许链式交接，但有次数上限，防止互相点名导致无限链条。
- **尽量不误触发**：例如代码块、引用里的 `@xxx` 会被跳过；非行首点名在 strict 模式下默认不转派（更保守）。

## 1. 已实现机制总览（按消息方向）

### 1.1 入站（Telegram → 某 bot）

1) **群聊点名门禁（Mention gating）**（偏系统机制）
- 配置为 `MentionMode::Mention` 时：群里没点名 bot ⇒ `AccessDenied::NotMentioned`。
- 该分支不会触发 LLM，但会进入 **listen-only ingest-only** 入库（见“场景 A”）。

2) **自我点名清理（strip self mention）**（偏会话文本协议）
- 触发 LLM 的入站消息：在入库/送 LLM 前，删除 `@本 bot`（见“场景 B”）。
- listen-only ingest-only 的入站消息：也会做同样的 self-mention stripping（见“场景 B”）。

3) **群聊命令只认定向（/cmd@bot）**（机制 + 协议都有）
- 群/频道里，`/cmd`（未带 `@this_bot`）会被当作“多 bot 歧义”而忽略，避免刷屏（见“场景 C”）。

### 1.2 出站（某 bot → Telegram 群）后的“写回其它 bot”

4) **mirror：把 A 的出站回复写入 B/C 的 session**（机制 + 协议都有）
- 写入文本前缀形如：`[@botA mirror] ...`（见“场景 D”）。
- 只写入“已存在”的 (bot, group) session，避免创建“幽灵群会话”。

5) **relay：把 A 出站里的 `@botB ...` 转成 B 的入站触发**（机制 + 协议都有）
- B 实际收到的文本前缀形如：`（来自 @botA）...`（见“场景 E”）。
- 支持“链式转派”与 hop 限制，避免无限 relay。

6) **media mirror 占位**（机制 + 协议都有）
- 目前只对“截图”出站做了 V1 占位镜像：`（发送了一张图片）`（见“场景 F”）。

## X. rewrite 示例速查表

> 目的：把“群聊场合下，消息到底被怎么改写/注入”的常见场景做成速查表。  
> 口径：只描述 **as-is**（当前代码行为）；示例里“写进 session / 送 LLM 的文本”指目标 bot 真正看到的 `content`。

### X.1 入站 rewrite（人 → bot）：self-mention stripping / 空白归一化

1) **单点名：删除 `@本 bot`**

群里原文：
```text
@c 请执行 X
```

bot C 写进 session / 送 LLM：
```text
请执行 X
```

2) **多点名：每个 bot 只删自己的那个 `@`（因此各自看到的文本不一致）**

群里原文：
```text
@a @b @c 请执行 X
```

- bot A：
```text
@b @c 请执行 X
```
- bot B：
```text
@a @c 请执行 X
```
- bot C：
```text
@a @b 请执行 X
```

3) **只有一个 `@本 bot`：strip 后为空 → 不触发 LLM，走固定短回复**

群里原文：
```text
@c
```

bot C strip 后文本为空（无附件时）：
- 不跑 LLM
- 直接返回固定短句（例如“我在。”）

4) **空白归一化：可能压掉换行/多空格（结构化文本可读性下降）**

群里原文：
```text
@c 你做一下：
- A
- B
```

bot C 写进 session / 送 LLM（示意）：
```text
你做一下： - A - B
```

### X.2 入站 rewrite（群聊 slash）：只认 `/cmd@this_bot`

群里原文：
```text
/help
```
bot C：忽略（避免多 bot 歧义刷屏）。

群里原文：
```text
/help@c
```
bot C：会处理（并会剥离命令里的 `@c` 后缀，避免把自我点名当正文噪声）。

### X.3 出站注入（bot A → 其它 bot）：mirror / relay

1) **mirror：把 bot A 的出站回复写进其它 bot 的 session（旁观记录，不触发推理）**

群里可见（bot A 发出）：
```text
结论：可以按方案推进。
```

bot B 的 session 里追加：
```text
[@botA mirror] 结论：可以按方案推进。
```

2) **relay：bot A 文本里出现“派活式 @botB …” → 注入给 bot B 并触发推理（B 回复会 reply 到 A 那条消息下）**

群里可见（bot A 发出）：
```text
@b 请把这个 PR 的风险点列一下。
```

bot B 被注入并看到：
```text
（来自 @botA）请把这个 PR 的风险点列一下。
```

3) **relay（Strict）：非行首 @ 默认不算派活，不触发 relay（但 mirror 仍可能写入旁观记录）**

群里可见（bot A 发出）：
```text
我觉得 @b 的方案里 RootMap 这块还要补证据链。
```

默认 `relay_strictness = Strict`：
- 不触发 relay（不叫醒 bot B）
- 但这句话仍可能以 mirror 形式进入 bot B 的 session：
  - `[@botA mirror] 我觉得 @b 的方案里 RootMap 这块还要补证据链。`

4) **relay（Loose）：非行首 @ 会走一次 LLM 分类（directive 才触发；失败/不确定默认 reference）**

群里可见（bot A 发出）：
```text
麻烦 @b 帮我把风险点列一下
```

`relay_strictness = Loose`：
- 可能触发 relay（取决于分类器判定）

5) **relay 扫描会跳过：代码块 / 引用行 / 行内代码（防误触发）**

群里可见（bot A 发出）：
```text
> 示例：@b 请执行 X
```
或：
```text
你可以这样写：`@b 请执行 X`
```
或 fenced code：
```text
```sh
echo "@b do something"
```
```

结果：这些 `@b` 不参与 relay 触发。

6) **跨行“点名 + 任务”：task_text 不要求同一行（只要后面还能抽到非空文本）**

群里可见（bot A 发出）：
```text
@c

你处理下 X
```

当前实现里，relay 抽取到的 `task_text` 仍可能是：
```text
你处理下 X
```
因此在 Strict 下也可能触发 relay（因为 `@c` 在某一行行首，且任务文本非空）。

7) **media mirror 占位：媒体出站会以占位文本镜像进其它 bot 的 session**

bot A 出站发图（群里可见为图片本体）后，bot B session 里追加（示意）：
```text
[@botA mirror] （发送了一张图片）
```

## 2. 场景 A：群里没点名 → listen-only ingest-only（不跑 LLM）

> 类型：偏**系统机制**（决定“触不触发 LLM/回不回群”）；同时会把消息写进会话历史（写进去的文本是否被改写属于“会话文本协议”，见场景 B）。

### 2.1 发生条件（人话）

- 你在群里说话，但**没有点名某个 bot**（并且该 bot 的配置要求必须点名）。
- 该 bot 侧不会回复，也不会触发推理；但会把消息写进它自己的群聊 session，用于“让它知道群里发生过什么”。

### 2.2 例子

群消息原文：
- `大家先同步一下今天的进度。`

bot C（未被点名）在本地 session 里会发生什么：
- 入库：会写入这条文本（标记 `ingestOnly: true`）。
- 不会发生：不会触发 LLM、不会在群里发消息。

## 3. 场景 B：自我点名清理（strip self mentions）+ 空白归一化

> 类型：偏**会话文本协议**（写进会话/送进模型的纯文本长什么样）。

### 3.1 发生条件（人话）

- 这条入站消息会进入某个 bot 的 session（无论是触发 LLM 的正常入站，还是 listen-only ingest-only）。
- 如果消息里出现了 `@这个 bot`（通过 Telegram entities 或文本 fallback 识别），就会把这一段删除。
- 删除后会做**空白归一化**：所有连续空白（含换行）会被压成单个空格。

### 3.2 例子：单点名

群消息原文：
- `@c 请执行 X`

进入 bot C 的 session / 送到 bot C 的 LLM 看到的文本：
- `请执行 X`

### 3.3 例子：多点名（你提到的“@我噪声”问题）

群消息原文：
- `@a @b @c 请执行 X`

各 bot 侧“自我点名清理”后的文本是**不一样的**：

| bot | 它最终看到的文本（入库/送 LLM） |
|---|---|
| a | `@b @c 请执行 X` |
| b | `@a @c 请执行 X` |
| c | `@a @b 请执行 X` |

这就是你担心的点：对 bot C 来说，文本看起来像“只在对 a、b 说话”。当前实现确实是这样（只删 self，保留其它 `@`）。

### 3.4 例子：只发了一个 `@bot`（无其它内容）

群消息原文：
- `@c`

bot C 侧：
- self-mention stripping 后文本为空；
- 如果没有附件，会直接回一个固定短句：`我在。`（不触发 LLM）。

### 3.5 例子：带定向命令的 `@bot` 后缀也会被剥离

群消息原文：
- `/help@c`

bot C 侧会把 `@c` 这个后缀从命令里剥离（避免命令文本带上自我点名）。

## 4. 场景 C：群聊 Slash 命令只认定向（避免多 bot 歧义）

> 类型：机制 + 协议都有（系统决定“要不要处理/要不要回群”；同时会对命令文本做一致性处理）。

### 4.1 规则（人话）

- 在群里：只有 `/cmd@this_bot` 会被这个 bot 处理。
- `/cmd`（未定向）会被忽略，即便该 bot 的 `mention_mode` 很宽松也一样。

### 4.2 例子

群消息原文：
- `/help`

bot C：
- 不处理（忽略）。

群消息原文：
- `/help@c`

bot C：
- 处理并回复帮助信息。

## 5. 场景 D：mirror（把 bot A 的出站回复写入其它 bot 的 session）

> 类型：机制 + 协议都有（系统决定“写不写入其它 bot 的会话”；写进去的文本前缀属于协议）。

### 5.1 发生条件（人话）

- bot A 在一个**Telegram 群**里成功发出了文本/语音字幕回复。
- 网关会把这条内容写入其它 bot（B/C/...）在同一个群的 session 历史里，前缀标记来源：
  - `[@botA mirror] ...`
- 但只会写入那些**已经存在**的 (bot, group) session，避免误创建“幽灵群会话”。

### 5.2 例子

群里实际可见（bot A 发出）：
- `结论：可以按方案推进。`

bot B 的 session 历史里新增一条（只入库，不在群里再发一次）：
- `[@botA mirror] 结论：可以按方案推进。`

说明：
- mirror 主要用于“让其它 bot 知道 A 说了什么”，本身不触发“被动执行”。

## 6. 场景 E：relay（从 bot A 出站文本里提取 `@botB ...` 并触发 B）

> 类型：机制 + 协议都有（系统决定“触不触发另一个 bot”；注入给目标 bot 的那条文本前缀属于协议）。

### 6.1 发生条件（人话）

- bot A 在 Telegram 群里发出的**出站文本**中包含对其它 bot 的 `@mention`。
- 网关会扫描出站文本（会跳过代码块、引用行、inline code，减少误触发），把其中被判断为“派活”的 `@botB 任务` 转成 bot B 的入站触发。
- bot B 看到的文本会带来源前缀：
  - `（来自 @botA）任务...`
- bot B 的 reply target 会被绑定到 **bot A 那条出站消息**，所以 B 在群里的回复会“挂在”A 那条消息下面，形成可追溯链路。

### 6.2 例子：行首点名（一定 relay）

bot A 出站文本：
- `@b 请把这个 PR 的风险点列一下。`

bot B 实际被触发的入站文本：
- `（来自 @botA）请把这个 PR 的风险点列一下。`

bot B 在群里的回复形式：
- 会 reply 到 bot A 的那条消息下（而不是凭空发新楼）。

### 6.3 例子：非行首点名（Strict 模式默认不 relay）

bot A 出站文本：
- `我觉得 @b 的方案里，RootMap 这块还要补证据链。`

在默认 `relay_strictness = Strict` 时：
- 不会 relay（避免“被引用/举例”导致误触发）。

在 `relay_strictness = Loose` 时：
- 会尝试让 LLM 把该 mention 标成 `directive|reference`；
- 标成 `directive` 才 relay；解析失败/调用失败则不 relay（保守）。

### 6.4 例子：代码块/引用行不会参与 relay 扫描

bot A 出站文本：
- ```text
  ```sh
  echo "@b do something"
  ```
  ```

或：
- `> 示例：@b 做 X`

结果：
- 不会 relay（防误触发）。

## 7. 场景 F：media mirror 占位（当前只做截图）

> 类型：机制 + 协议都有（系统决定“是否做媒体占位镜像”；占位文本内容属于协议）。

### 7.1 例子

bot A 出站给群里发了一张截图（媒体消息）：
- 群里可见的是图片本体。

bot B 的 session 历史里会新增一条占位 mirror（只入库）：
- `[@botA mirror] （发送了一张图片）`

## 8. 已知问题（现状系统机制 + 现状会话文本协议）

> 注意：下面这些问题有的源自“系统机制”（例如冷启动、去重、循环），有的源自“会话文本协议”（例如 self-mention stripping 导致多视角不一致、mirror 前缀易误读）。

1) **多点名时，每个 bot 看到的文本不一致**
- `@a @b @c 请执行 X` 在 c 侧会变成 `@a @b 请执行 X`，确实容易让 c 误读“好像没叫我”。
- 但它的本意是：避免把“@我”噪声带进 LLM；实现上只做了最小动作（删 self）。

2) **空白归一化会压掉换行/多空格**
- 如果用户用换行做结构化（例如列表），strip 后会变成一行，可能降低可读性。

3) **mirror/relay 都不会创建目标 bot 的新群 session**
- 如果 bot B 从未在这个群建立过 session，那么 mirror/relay 都会跳过（避免“把不在群里的 bot 误当成在群里”）。

4) **冷启动“看不见/叫不动”：bot 明明在群里，但第一次协作会断链**
- 现状的安全策略是“宁可不镜像/不转派，也不要创建幽灵会话”，所以需要目标 bot 的 `(bot, group)` session 先存在。
- 例子：`@pm_bot` 先在群里给 `@coder_bot` 派活，但 `coder_bot` 从未在该群建立过 session：
  - mirror：`pm_bot` 的回复不会写进 `coder_bot` 的会话历史；
  - relay：即使 `pm_bot` 明确写了 `@coder_bot 请做 X`，也可能因为目标会话不存在而无法触发 `coder_bot`。

5) **去重不够稳：重试/重启后可能重复触发（成本 + 噪声）**
- 现状有去重，但口径分散（例如 mirror 主要靠“看会话尾巴里是否已有 mirrorKey”，relay 有进程内去重）。
- 例子：网关重启或出站链路发生重试后，同一条“转派/转述”可能再次执行：
  - relay 重复：`coder_bot` 被重复叫醒跑两次，群里出现两份重复结果；
  - mirror 重复：其它 bot 的会话里出现多条重复的 `[@botA mirror] ...`，上下文变厚、可读性变差。

6) **防循环不彻底：容易出现“踢皮球”式互叫**
- 有 hop limit 可以挡住“一条链走太长”，但挡不住 A→B、B→A 这种反复生成新链的来回转派。
- 例子：
  - `pm_bot` 转派：`@coder_bot 给我一个结论`
  - `coder_bot` 回应里又写：`@pm_bot 你先把背景贴一下`
  - 如果双方输出都持续用 `@对方 ...` 这种写法，系统可能不断触发新的 relay，造成反复唤醒与刷屏风险。

7) **Telegram 前置设置导致“消息根本进不了系统”：代码层无法解决，只能记录与自检**
- 如果 Telegram 没把“未点名消息”投递给某个入口（各 bot 或 sentinel），系统就没有这条消息的事实记录。
- 例子：群里有人发了一个“交付消息”（没 @ 任何 bot），你期待所有 bot 的会话里都能旁听到；但如果平台没投递，这条消息在系统里就不存在，后续也谈不上不断链/追赶。

8) **谁说的话，LLM 不一定看得到：元信息在系统里，但不一定进入模型上下文**
- session 里会保存 `username/senderName` 等 channel 元信息；但当前 LLM 上下文构建主要使用纯文本 `content`，不会自动把这些元信息变成“模型能读懂的聊天记录署名”。
- 例子：群里同一句“这个可以吗？”来自 Owner 还是普通成员，对模型的决策意义完全不同；如果文本里没有署名/角色提示，模型可能做出不符合权限预期的回应。

9) **转述（mirror）容易被模型误当成“有人对我下指令”**
- 现状 mirror 是把别的 bot 的出站内容写成 `role=user` 的会话输入，再加一个前缀 `[@botA mirror] ...`。
- 例子：`pm_bot` 只是对群里同步了一段结论，`coder_bot` 的会话里会出现一条“用户消息”形式的转述；模型可能把它当成“有人对我说：请按这个结论执行”，从而误触发动作或误解上下文。

## 8.5 当前讨论中的“建议处理方式”（先记口径，后续再细化）

> 这段记录的是**当前讨论里你提出的处理取向**（不等于已经实现）。

1) **冷启动/会话激活：优先用“系统侧建群配置/入群名单”自动化解决**
- 取向：不靠人工在群里手工输入命令来“接入”；系统应有一份“这个群有哪些 bot 参与”的配置（roster），据此允许自动建档/激活会话，避免幽灵会话。
- 例子：群 `DevGroup` 的 roster 明确为 `{pm_bot, coder_bot, qa_bot}`。系统只会为这三只 bot 自动创建/激活该群会话；其它本实例 bot 不会被误拉进来。

2) **去重与重启一致性：初期倾向“强提示 + 先忽略”，避免上来就做重资产落盘**
- 取向：如果重复触发的影响可控，优先把现状“记录清楚 + 可观测 + 提示明显”，先不做复杂持久化去重。
- 例子：重启后偶发一次重复 mirror，先接受；但需要让排障者能一眼看出来“这是重复写入/重复触发”。

3) **防循环/踢皮球：初期倾向用“文本协议 + bot prompt/层级制度”约束，而不是加复杂系统机制**
- 取向：暂不处理，或者是用明确的协作层级来降低互相转派（例如 PM 可以派活给员工；员工不应互相派活，必要时回给 PM 再派）。
- 例子：`coder_bot` 需要 `qa_bot` 支持时，不直接输出 `@qa_bot ...` 去触发；而是回给 `pm_bot`：“我需要 QA 做 X”，由 `pm_bot` 再统一派活。

4) **Telegram 平台前置条件：代码层无解，先记录与自检**
- 取向：把 privacy mode/权限/投递前置写成 checklist；上线前通过简单群内自检确认“消息确实进系统”。
- 例子：群里发一条不点名的测试消息，确认每个 bot 的会话历史里都能看到 ingest-only 记录；看不到就先修 Telegram 设置，而不是在代码里硬补。

5) **权限系统（谁能叫谁/谁的指令更高优先级）：现阶段先不做**
- 取向：先把协作跑通，权限边界后续再加固。

6) **“谁在说话 / 是否在叫我”：后续通过会话文本协议解决**
- 取向：不指望 LLM 自动读取 session 的 channel 元数据；需要把“说话人/是否点名”等关键信息用简洁文本模板体现在会话消息里。

> 说明：第 2～7 节是“现状机制/现状协议”的事实梳理；第 8 节是已知问题；第 8.5 节是当前讨论中的取向记录（还未落地）。
