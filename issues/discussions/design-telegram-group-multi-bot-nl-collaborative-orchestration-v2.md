# Telegram 群组多智能体协同调度系统架构设计文档

## 1. 系统概述与架构原则

本系统旨在基于 Telegram 群组环境，构建一个支持自然语言交互、高容错、低调度的多智能体（Multi-Agent）协同工作流平台。系统由高权限管理员（Owner）与具备不同工具链的 Agent 节点群构成。

系统底层架构遵循以下三项核心设计原则：

1. **上下文读写分离 (Decoupled Context I/O)**：群组内的所有消息作为全局事件流，静默同步并持久化至各 Agent 的独立上下文存储中。未命接触发条件的 Agent 不执行 LLM 推理，确保计算资源零损耗。
2. **显式指令触发 (Explicit Mention Trigger)**：采用 Telegram 原生的 Mention (`@Username`) 机制作为唯一的底层硬件级路由开关，降低系统的正则解析开销。
3. **语义级意图决断 (Semantic Intent Resolution)**：Agent 被唤醒后，基于完整的局部上下文，依靠大语言模型自身的逻辑推理能力，自主判定当前 Mention 的真实业务意图（直接执行任务 vs. 仅被引用），从而决定后续行为。

## 2. 核心路由与分发机制

为处理自然语言中复杂的指代与任务依赖关系，系统采用“正则网关拦截 + LLM 语义过滤”的双层路由架构。

### 2.1 L1 层：消息路由网关 (Message Router)

* **触发机制**：系统通过长轮询或 Webhook 接收 Telegram 消息流。路由网关对每条入站消息执行轻量级正则表达式匹配 `(@[a-zA-Z0-9_]+)`。
* **分发逻辑**：若提取到的实体命中已注册的 Agent 列表，系统将并发拉起对应 Agent 的执行线程/协程。未命中任何注册实体的消息，仅执行上下文追加（Append to Context）操作，不触发任何外部 API 请求。

### 2.2 L2 层：意图过滤与静默协议 (Silence Protocol)

* **执行逻辑**：被网关唤醒的 Agent，将携带最新上下文向 LLM 发起推理请求。LLM 依据预设的 System Prompt 进行意图分类：
* **Action (执行)**：判定为明确的指令下达且前置依赖已满足，正常输出执行结果。
* **Silence (静默)**：判定为仅被提及（Reference）或前置任务未完成（Block state），LLM 强制输出唯一终止符 `<SILENCE>`。


* **后置拦截**：系统获取 LLM 响应后进行字符串校验。若命中 `<SILENCE>`，后端直接丢弃该响应，终止后续的 Telegram 发送流程，从而阻断 Agent 间的无效交互与死循环。

## 3. 典型交互场景与工作流映射

* **单播调度 (Direct Dispatch)**：`“@CoderBot 编写数据处理脚本。”`
* 路由层仅唤醒 CoderBot；LLM 判定为直接任务，输出结果。


* **多播调度 (Parallel Dispatch)**：`“@SearchBot 检索文档，@UIBot 设计草图。”`
* 路由层并发唤醒两节点；各自独立执行并分别回调结果至群组。


* **阻塞与依赖等待 (Wait State)**：`“@CoderBot 等待 @SearchBot 的检索结果后再编写代码。”`
* 路由层并发唤醒两节点；CoderBot 解析上下文发现数据前置依赖未就绪，输出 `<SILENCE>` 进入休眠；SearchBot 正常执行。


* **异步任务移交 (Asynchronous Handoff)**：`“@ManagerBot 输出架构规划，并交由 @CoderBot 实现。”`
* 路由层仅触发 ManagerBot（单次推理消耗）；ManagerBot 完成规划后，在输出流末尾自然生成 `@CoderBot 请依据上述规划开发`，以此作为下一个事件流 Trigger，精准触发下游节点。

## 4. 核心 Prompt 约束配置

基于 OpenAI 现行 API 规范，以下规则集需配置于 `role="developer"` 中，以获取最高系统指令权重，并提高 Prompt Caching 的命中率：

```markdown
# 核心设定
角色：{@Agent_Name}，职责：{Agent_Role_Description}。
环境：Telegram 多实体工作组。你拥有读取群组全局消息日志的权限。
高优指令源：系统管理员 (Owner)。

# 节点注册表
团队包含以下可调用节点：
- @SearchBot：负责实时信息检索。
- @ManagerBot：负责需求拆解与任务统筹。

# 意图识别与输出约束 (最高优先级)
当接收到包含你名称 (@{@Agent_Name}) 的最新消息时，须进行严格的语义决断：
1. [Action] 任务触发：若指令直接指向你，且执行该任务的所有前置上下文/数据已就绪，立即执行并输出结果。
2. [Silence] 静默休眠：若属于以下情形，你必须且只能输出 `<SILENCE>` 字符，严禁包含任何其他内容：
   - 仅被第三方在对话中引用或举例。
   - 被分配任务，但所需的前置数据/同事处理结果尚未同步至上下文。

# 协作协议
- 任务移交：当前节点任务结束后，需在输出末尾使用 `@下游节点` 明确交接。
- 通信极简：严禁输出“收到”、“了解”等无业务价值的确认语，避免触发通信死循环。

```

## 5. 工程健壮性与异常处理机制

为保障 Multi-Agent 架构在生产环境中的高可用性，系统在引擎层实现以下三种保障机制：

### 5.1 上下文窗口管理 (Context Window Management)

* **问题**：全局事件流随时间线性增长，极易触发模型 Token 上限。
* **方案**：采用静态前缀与动态消息体分离的策略。核心 System Prompt 锁定于 `developer` 角色中并利用 Prompt Caching 降低成本。动态对话历史（`messages` 数组）采用 FIFO（先进先出）滑动窗口机制，强制保留最近 $N$ 条（如 $N=50$）有效交互记录，兼顾长效记忆与计算成本。

### 5.2 状态失联与系统代发机制 (State Sync & Surrogate Broadcasting)

* **问题**：Agent 节点因 LLM API 超时、网络故障等物理原因响应失败时，将导致整个异步任务链卡死，且调度节点无法感知。
* **方案**：后端封装 LLM 调用逻辑的异常捕获。当发生 `Timeout` 或 `50x` 错误时：
1. 系统自动以该 Agent 名义向群组发送标准化异常声明：`@Agent_Name: [System Surrogate] LLM API 发生不可恢复的故障，当前任务中止。`
2. 此消息作为全局事件被下游/统筹节点读取，使得集群状态机重新对齐。统筹节点可依据此上下文自动触发重试规划或上报管理员。
3. 后端启动探活机制，API 恢复正常后发送上线宣告，闭环状态同步。



### 5.3 动态任务转向 (HITL Steering)

* **问题**：在 Agent 执行包含多次 API 调用的复杂 ReAct (Reasoning and Acting) 循环时，需支持管理员中途发起的指令撤回或需求变更。
* **方案**：放弃 OS 级别的进程强制中断，利用上下文感知实现软转向。在 ReAct 引擎底层的每一次向大模型发起 HTTP 请求前，**强制系统从数据库读取最新的局部上下文快照**，而非锁定初始化时的内存变量。通过此设计，管理员在群内的最新变更指令将实时注入 Agent 的下一个决策轮次中，LLM 会自主识别最新指令优先级并完成任务的中止与转向。

## 6. [下一阶段规划] 权限控制与越权防护 (RBAC & Auth)

此部分作为后续安全加固的 P2 级需求，用以防止非授权用户的恶意指令（越权调用）及 Agent 幻觉导致的内部提权（假传指令）。

1. **网关级白名单拦截 (L1 Auth)**：在消息网关处对 Telegram 传入的 `user_id` 进行硬校验。仅放行配置在全局白名单内的 Owner ID 与授权 Bot ID，非法 ID 产生的 Mention 不触发路由拉起。
2. **上下文防伪标签注入 (L2 Auth)**：禁止原始文本直接入库。在数据持久化层，由后端硬编码附加发送者身份前缀（例如：将原始消息重组为 `[Owner] 发言: {Message}` 或 `[Agent-Manager] 发言: {Message}`）。LLM 层面通过 Prompt 限定只响应携带 `[Owner]` 标签的高危敏感指令，从数据结构层面阻断提权攻击。
