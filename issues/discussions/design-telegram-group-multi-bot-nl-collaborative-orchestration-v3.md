# Telegram 多智能体自然语言协同系统设计方案 (V3)

## 1. 系统背景与核心目标

在 Telegram 群组环境中，通过“后台调度网关 + 智能体协议”实现多 Agent（机器人）的拟人化协同。

* **核心目标**：极简交互（纯自然语言）、极低成本（精准唤醒）、逻辑闭环（任务不断链）。
* **核心组件**：
* **调度后台（传达室大爷）**：负责物理拦截消息、维护“挂起登记表”、执行唤醒算法。
* **独立上下文数据库**：为每个 Bot 记录全量群聊历史，确保唤醒时具备完整记忆。
* **状态化静默协议**：Bot 通过特定暗号告知后台其当前状态（等待或忽略）。



---

## 2. 核心运行规则：三路归并唤醒算法

这是确保任务不断链的物理层核心。当群内产生新消息 **N**（若 N 回复了旧消息 **M**）时，后台调度器计算以下三个集合的并集：

* **路向 A (显式唤醒)**：消息 N 文本中直接 `@Username` 提到的 Bot。
* **路向 B (定向回复)**：若消息 M 的作者是某个 Bot，则唤醒该 Bot（处理追问/反馈）。
* **路向 C (溯源唤醒)**：查找“挂起登记表”，所有在消息 M 下状态为 `WAITING` 的 Bot。

**最终动作**：对上述去重后的 Bot 集合执行并行 LLM 唤醒，并排除消息 N 的发送者本人。

---

## 3. 智能体响应协议 (Silence Protocol)

Bot 被唤醒后，其输出必须符合以下三类规范之一，由后台进行拦截处理：

1. **【执行内容】**：正常的工作成果、计划或回复。由后台直接转发至 Telegram 群聊。
2. **`<SILENCE:PASS>`**：意图判定为“纯提及”或“与己无关”。后台直接丢弃，群内保持静默。
3. **`<SILENCE:WAIT>`**：意图判定为“任务属于我，但前置条件未就绪”。
* **后台动作**：丢弃该响应，并在数据库记录一条挂起状态：`[消息M的ID] -> [Bot名字] -> STATUS:WAITING`。



---

## 4. 全场景矩阵与闭环逻辑验证

| 场景分类 | 典型交互场景描述 | 调度逻辑（后台视角） | 闭环验证（为什么不断链） |
| --- | --- | --- | --- |
| **标准派活** | Owner: `@bot1` 写个脚本。 | 触发路向 A。唤醒 bot1。 | **已闭环**：bot1 直接输出成果。 |
| **显式接力** | Owner: `@bot1` 查资料，`@bot2` 随后编码。 | 并发唤醒两人。bot1 执行；bot2 发现没资料，返回 `WAIT` 并登记。 | **已闭环**：待 bot1 **回复**该指令并提交资料后，触发路向 C，自动踢醒 bot2。 |
| **总包模式** | Owner: `@bot1` 统筹，让 bot2 实施（未艾特 bot2）。 | 仅唤醒 bot1。bot2 持续休眠以节省成本。 | **已闭环**：bot1 完工后根据“接力协议”在回复中手动 `@bot2`。 |
| **定向追问** | Owner 回复 bot1 的消息：“改一下逻辑”。 | 触发路向 B。精准唤醒 bot1。 | **已闭环**：bot1 针对反馈内容进行修正。 |
| **断链补救** | bot1 提交了资料，但由于幻觉忘了艾特 bot2。 | 只要 bot1 点击了 **Reply（回复）** 原指令发报。 | **已闭环**：触发路向 C。后台查表发现 bot2 在等这把“钥匙”，强行踢醒 bot2。 |
| **动态转向** | 任务中途，Owner: `@bot1` 停止 A，改做 B。 | 后台更新上下文。 | **已闭环**：bot1 在下一轮推理间隙拉取到新指令，实现物理转向。 |
| **物理故障** | bot1 接口超时或崩溃。 | 后台 Catch 异常。 | **已闭环**：后台代发故障公告，Manager 节点或 Owner 介入对齐信息，防止死等。 |
| **安全防御** | 非白名单路人甲艾特 `@bot1` 删库。 | 后台校验 `user_id`。 | **已闭环**：校验失败，不予唤醒，Bot 维持静默且零消耗。 |
| **噪声控制** | 闲聊提到 “上次 `@bot1` 做的不错”。 | 触发路向 A。唤醒 bot1。 | **已闭环**：bot1 判定为“纯引用”，回 `PASS`。群内静默，不干扰对话。 |
| **自唤醒防御** | bot1 幻觉发送：“我是 `@bot1`”。 | 正则发现 `@对象` = 发送者。 | **已闭环**：网关直接过滤，不予触发，杜绝死循环。 |

## 5. 核心 System Prompt 约束 (Bot 指令集)

为确保逻辑闭环，所有 Bot 必须在 `role="developer"` 中固化以下协议：

> **1. 意图自检协议**：
> * 若指令指向你且条件具备 -> 立即执行。
> * 若指令指向你但你在等待他人产出 -> 输出 `<SILENCE:WAIT>`。
> * 若你仅被引用或任务不属于你 -> 输出 `<SILENCE:PASS>`。
> 
> 
> **2. 隐式任务接力**：
> * 若 Owner 指令提到了其他同事但未显式艾特，作为当前任务负责人，你必须在完工时手动 `@该同事` 移交权限。
> 
> 
> **3. 链路维护规范**：
> * 提交反馈或资料时，**必须使用 Telegram 的回复 (Reply) 功能**针对原始指令进行回复。
> 
> 
> **4. 社交沉默原则**：
> * 严禁任何社交冗余（如“收到”、“谢谢”）。除工作成果外，仅允许输出预设的静默信号。
> 
>

## 6. 工程稳定性保障机制

1. **脑容量管理**：核心协议锁定在 `developer` 角色（利用缓存）；动态聊天历史采用 FIFO 滑动窗口截断，确保永远不跑爆上下文窗口。
2. **监工与自愈**：后台程序执行“心跳探活”。若 Bot 掉线，后台代发故障声明；若 Bot 康复，后台代发回归声明。Manager 节点根据此类“系统消息”自动重拟计划。
3. **循环间隙转向**：Bot 执行 Loop 时，每一轮新的 LLM 请求前，后台强制重新拉取全局上下文数据库，确保 Owner 的插话指令（Steering）具有最高即时性。

---

## 7. 下一阶段工作提示：权限安全 (RBAC)

* **物理白名单**：调度网关仅响应 `OWNER_ID` 和 `BOT_IDS` 产生的唤醒信号。
* **身份标签注入**：后台在消息存入数据库前强制贴标（如 `【最高指挥官】说：...`），Bot 只认标签不认人名，从逻辑上杜绝“假传圣旨”。
