# Telegram 群组多智能体（Bot）自然语言协同调度方案 (最终完整版)

## 一、 系统背景与核心理念

在 Telegram 工作群中，包含一位最高权限的 Owner（您本人）和多位分工不同的 AI Bot。本系统的终极目标是打造一个**完全基于自然语言、无需生硬指令、高容错、低成本**的虚拟数字人协同办公环境。

系统遵循三大“铁律”：

1. **全员静默旁听（独立上下文）**：群内所有发言均被后台默默记入每个 Bot 的独立数据库中。在未被明确触发时，Bot 绝对不消耗大模型 API。
2. **名字即物理开关（艾特触发）**：后台“传达室大爷”（正则匹配程序）只拦截带有 `@Bot名字` 的消息，并仅唤醒被点名的 Bot 进行思考。
3. **听话听音（语义决断）**：被唤醒的 Bot 结合自身的完整上下文自行判断：是被直接派单（立刻干活），还是仅仅被引用/条件未成熟（输出 `<SILENCE>` 暗号，后台静默丢弃）。

---

## 二、 核心机制：“物理把关 + 脑力决断”

* **第一层：物理把关（极低成本）**：群消息到达后台，只寻找 `@` 符号。扫到谁，就单独拉起谁的 LLM 进程。没扫到的 Bot 继续静默记笔记。
* **第二层：脑力决断（语义防呆）**：大模型阅读上下文。如果判定需要干活，正常输出；如果判定自己只是被“引用”或前置任务没做完，强制输出 `<SILENCE>`，后台拦截该输出，不发往群聊，彻底杜绝多 Bot 乱抢话和死循环。

---

## 三、 交互场景全面推演

* **单点直派**：`“@CoderBot 写个脚本。”` 👉 仅 CoderBot 醒来干活。
* **平行分发**：`“@SearchBot 查资料，@UIBot 画原型。”` 👉 两人同时醒来，各自独立干活。
* **无效触发/预告**：`“@CoderBot 等 @SearchBot 查完资料你再写代码。”` 👉 两人均被唤醒。SearchBot 查资料；CoderBot 发现条件未满足，输出 `<SILENCE>` 闭嘴。
* **总包接力（极其优雅）**：`“@ManagerBot 你出计划，然后让 @bot2 去写代码。”` 👉 仅 ManagerBot 消耗推理，并在其回复末尾自然生成 `@bot2 请开始写代码`，从而触发下一环。
* **遇到困难求助**：Bot 工作中途遇到报错，直接在群里 `@SearchBot 帮我查一下`，拿到群里的回复结果后继续自己的工作 Loop。

---

## 四、 核心 System Prompt 模版

利用大模型能力解决“意图判断”的核心，在每次请求时，将此模版作为 `developer` (或 `system`) 角色注入最高权重：

```markdown
# 身份与环境
你的名字是 {@CoderBot}。身份是：{资深程序员}。
你身处 Telegram 工作群，能看到所有历史消息（即你的上下文）。群里的 Owner 是最高指挥官。

# 团队花名册
你可以随时呼叫同事协助：
- @SearchBot：负责联网查阅最新资料。
- @ManagerBot：负责统筹规划。

# 核心社交规则与意图判断（最高指令）
当你在最新消息中被 `@` 提及时，必须结合上下文分析意图：
1. 【做事（Action）】：对方直接向你下达指令，且前置条件已具备 👉 立刻执行任务并输出结果。
2. 【闭嘴（Silence）】：如果对方只是顺嘴“引用”你，或者给你派活但“前置条件未满足”，你必须立刻停止工作，且只能输出 `<SILENCE>` 这九个字符，绝不可附带任何废话。

# 派单与沟通礼仪
1. 自然派单：需要下一位同事接手时，在回复末尾用自然语言 `@` 他。
2. 严禁废话：绝对禁止说“好的”、“收到”、“谢谢”等客套话，防刷屏死循环。

```

## 五、 工程落地与三大核心盲区攻坚机制

在真实服务器环境中，系统依靠以下三种机制确保永不宕机、永不卡死、高度听指挥：

### 1. 脑容量管理：Developer 锁定 + 动态截断

* **机制**：群聊记录无限增长会导致 Token 撑爆。系统采用“读写分离”的截断机制。
* **落地**：将核心 Prompt 置于 `role="developer"` 中，利用 **Prompt Caching（提示词缓存）** 大幅降低每次唤醒的成本和延迟。对于群内动态聊天记录，后台采用简单的 **FIFO（先进先出）滑动窗口**（例如仅保留最近 50 条），确保永远不触发超长报错。

### 2. 死锁自愈：系统替身代发与状态感知 (Fail Fast & Loud)

* **机制**：解决 Bot 的大模型 API 突然超时报错，导致任务链断裂、全群卡死的问题。
* **落地**：
* **故障明报**：后台捕获 API 异常后，直接在群里以该 Bot 名义发告警：`@Coder_Bot：[系统替身代发] 我的大脑接口发生故障，失去响应。` 包工头 Bot 看到上下文更新，会自动接管异常，通知 Owner 或重新调度。
* **探活恢复**：后台开启静默 Ping 测试。一旦接口恢复，替身再次在群内发报：`@Coder_Bot：[系统替身代发] 故障已排除，我已恢复在线。` 完美实现 Multi-Agent 状态机在自然语言层面的同步。



### 3. 动态转向与中途插话 (Steering / Human-in-the-Loop)

* **机制**：解决老板在 Bot 处理复杂任务（如执行多轮 Tool Calling 的 Loop）中途，突然反悔或修改需求的问题，拒绝暴力的“杀进程”。
* **落地**：系统基于 ReAct 循环。**后台引擎在执行每一个单一的 LLM Request 前的一毫秒，必须强制从数据库拉取全群最新上下文**。这样，Owner 随时插话的 `@` 指令能像针一样扎进 Bot 正在运转的 Loop 间隙中。Bot 读到最新上下文后，会依靠自身推理主动中止旧任务，平滑转向新任务。

## 六、 [下一阶段工作] 权限安全与防越权机制

*(注：本部分作为后续上生产环境前的 P2 级需求记录，当前开发阶段暂不实装)*

为了彻底杜绝“假传圣旨”（如外人进群捣乱，或 Bot 产生幻觉互相下达危险指令），系统将在物理和逻辑层增加两道极简防线：

1. **物理防线（白名单拦截）**：
后台路由根据 Telegram 原生的 `user_id` 进行校验。只有发件人是 `OWNER_ID` 或合法的 `BOT_IDS` 时，命中 `@` 符号才允许唤醒大模型。无权限路人的发言只记入上下文，绝不触发 API（防刷额度）。
2. **逻辑防线（不可伪造身份标签注入）**：
后台代码在将群消息存入数据库前，强制拼接防伪标签。
* 发件人为 Owner 👉 存为 `【最高指挥官】说：...`
* 发件人为 Bot 👉 存为 `【平级同事 XXX】说：...`
同时在 System Prompt 中升级安全协议：Bot 仅听从带有【最高指挥官】前缀的高优指令，并有权拒绝【平级同事】下达的涉密/危险操作，从而实现军工级的权限隔离。
