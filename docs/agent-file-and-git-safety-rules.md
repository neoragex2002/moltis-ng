# Agent 文件与 Git 安全规则

更新：2026-03-05

## 🎯 核心原则

默认允许正常增量编辑，但必须严格避免任何未经确认的**破坏性写入**行为。

最主要风险不在于文件是否被 git tracked，而在于：

> **整文件重写、覆盖本地未提交内容、或在 untracked 阶段发生误删/误覆盖。**

---

## 一、允许的操作（默认放行）

在用户要求修改、优化、修复等场景下，Agent 可以：

- 对相关文件进行**增量修改**
- 添加或补充内容
- 创建新文件
- 执行只读类 Git 操作（status、diff、log 等）

前提：

> 必须优先采用最小差异修改（minimal diff），并严格限定在用户意图范围内。

---

## 二、严格禁止的操作（需用户明确确认）

未经用户明确确认，Agent 不得执行任何可能导致内容丢失或不可恢复的操作，包括但不限于：

### 文件层面

- 删除文件或目录
- 清空或截断文件
- 整文件覆盖式重写
- 批量 rename / move
- 大范围结构性重写
- 覆盖用户已有但未提交的修改

### Git 层面

- `git reset --hard`
- `git checkout --`（覆盖工作区）
- `git clean`
- force push / 历史重写（rebase 等）
- 任何可能丢弃工作区修改的操作

---

## 三、关键文档特别约束（issues/ 等）

对以下“决策载体”目录保持更高谨慎：

- `issues/*`
- `docs/*`
- 或用户明确标记的关键文档

规则：

未经用户明确要求：

- 优先增量 Update，避免 Delete + Add 覆盖式改写
- 非必要不得整文件重写
- 修改应保持**最小必要范围**
- 不得顺手进行无关重排或风格性重写

---

## 四、Untracked 窗口期保护（关键风险点）

Agent 必须意识到：

> 新创建但尚未纳入 Git 跟踪的文件处于最高风险阶段。

因此：

- 不得随意覆盖或清空 untracked 的用户文件
- 对新建关键文档，应避免后续整文件重写
- 如操作可能扩大影响范围，应先询问用户

---

## 五、不确定即询问（强制）

当出现以下任一情况时，必须暂停并询问用户：

- 无法判断操作是否可能造成内容丢失
- 需要整文件级修改
- 需要可能影响工作区安全的 Git 操作
- 修改范围可能明显扩大

> 禁止基于猜测自行执行高风险操作。

---

## ✅ 设计目标（行为预期）

本规则旨在实现：

- 🟢 日常修改顺畅
- 🟢 小步增量安全
- 🟡 大改前会提醒
- 🔴 极难发生“闭眼清空 / 重写”事故
- 🔴 不会无意破坏用户工作区
