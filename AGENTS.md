# AGENTS.md

本文件定义当前仓库的项目级 Agent 执行规则，用来补充 `CLAUDE.md`中的工程规范。凡是编写 issue 文档、修改关键决策载体、或实施重构、修复时，都必须严格遵守这里的约束。

## 一、强制参考文档与模板

### 基础参考
- `CLAUDE.md` —— 工程约定与 Rust 开发规范
- `docs/agent-file-and-git-safety-rules.md` —— Agent 文件与 Git 安全规则

### Issue 文档模板
Issue 文档必须使用 `issues/template/` 下的模板：
- 单 issue：`issues/template/TEMPLATE-issue-single.md`
- 多 issue / 总览 / 审计：`issues/template/TEMPLATE-overall-multi.md`
- 增量更新指南：`issues/template/TEMPLATE-update-guide.md`
- 仅需求模板：`issues/template/TEMPLATE-requirements.md`

## 二、命名与契约
- 内部代码标识符使用 `snake_case`，例如 Rust 函数、变量、结构体字段
- 外部 JSON / RPC 字段使用 `camelCase`，例如 UI / API 契约
- 变量命名必须精炼、清晰、明了、贴切，做到见文知意；不得使用模糊、含混、易混淆或有歧义的命名
- 优先使用显式映射（如 serde rename、`json!` 中的 key），不要在同一层混用不同命名风格

## 三、核心原则（强制）
- **第一性原则**：先回到系统真实目标、真实边界、真实约束做判断；禁止被历史实现、临时补丁、表层现象牵着走。方案、文档、命名、测试都必须服务于根因闭环，而不是服务于维持旧表象。
- **不后向兼容原则**：对用户明确要求的 one-cut / 硬切换 / 严格重构任务，默认不保留 legacy 兼容、fallback、alias、shim、双写、双读、静默迁就；除非用户明确批准，否则不得自行加入兼容尾巴。
- **唯一事实来源原则**：同一类事实只能有一个唯一归属（owner）；不得并行保留多个同级事实源，也不得依赖“配置 + 数据库 + 运行时补丁”混源拼装真相。涉及 owner 收口时，必须在 issue 或实现中明确：谁是唯一写入口、谁是唯一生效来源、谁只是投影或缓存。
- **关键代码路径测试覆盖原则**：关键功能 / 修复必须覆盖关键主路径、关键边界、关键失败面，优先证明“核心行为已闭环、关键回归已冻结”；不得只补外围用例而放过核心路径，也不得用大量重复 case 代替关键路径覆盖。
- **边缘情况覆盖原则（从属）**：边缘情况必须被系统性盘点并冻结，但它的优先级**永远低于**第一性原则、不后向兼容原则、唯一事实来源原则、关键代码路径测试覆盖原则。若唯一 owner、主路径语义、one-cut 删除面、关键测试口径尚未定稿，则不得先为边缘情况发明额外概念、兼容分支、fallback 或大而全规则；边缘覆盖只能在主合同已经收敛后，用来补齐失败面、竞态面和验收闭环，不能反过来主导系统设计。

## 四、严格重构与 one-cut 执行规则（强制）

### 4.1 硬切换规则
- 凡是用户**明确要求**的硬切换 / one-cut / 严格重构操作，实施时不得保留 fallback、alias、compat shim、silent degrade 等尾巴。
- 这类任务默认**不考虑后向兼容**、**不做自动数据迁徙**、**不做自动 schema rename / 自动字段映射 / 自动目录回退读取**。
- 命中 legacy 输入、配置或持久化形状时，必须按严格标准处理：直接报错或强告警，并给出明确修复指引；不得“先兼容跑起来再说”。
- 不得为了“识别 legacy 并专门拒绝它”而额外引入新的特例探测、legacy guard、预判分支或兜底机制；优先删除 legacy 路径或默认语义，让系统在新口径下按正常主路径自然失败。
- 若你判断保留兼容尾巴是唯一合理方案，必须先停下并征求用户明确确认；未经确认不得自行加入兼容路径。

### 4.2 范围收敛与高内聚
- 对用户明确要求的重构或收口类任务，必须先做**范围收敛**与**职责切分**审查；在 issue 或计划未证明“边界清楚、最小闭环明确、测试面收敛”之前，不得直接铺开实现。
- 实施方案必须优先选择**高内聚、单点收口、最小闭环**的做法；能在现有边界内闭合的，就不得把复杂度扩散到更多模块、层次或抽象。
- 非绝对必要，不得新增跨渠道或跨模块的通用抽象、公共概念、总线、框架层或“为未来预留”的接口；尤其不得借修一个渠道或一个 issue 之机顺手做泛化设计。
- 代码改动必须避免离散、散乱；同一类规则应集中在单一职责位置，不得在多个文件 / 多层路径里各写半套逻辑。
- 若某项“顺手优化 / 顺手统一 / 顺手抽象”不是当前 issue 达成闭环的必要条件，默认不做；确有必要扩大范围时，必须先在 issue 中补齐理由、边界与受影响面，再实施。
- 评审与复审时，必须主动检查：
  - 是否引入了非必要概念膨胀
  - 是否把渠道专属复杂性错误外溢到 `core` / `gateway` / `common` 层
  - 是否存在同一规则分散在两处以上实现
  - 是否可以通过删除冗余路径、复用既有结构进一步收敛
- 文档、issue、测试也必须遵守同样的收敛原则：只覆盖当前闭环所需内容，不扩写无关设计，不堆无关 case。

## 五、文档与工作区安全（强制）
- 必须严格遵守 `docs/agent-file-and-git-safety-rules.md`
- 以下条目只是执行时自检摘要，不替代原文
- 默认只做最小增量修改（minimal diff），严格限定在用户意图范围内
- 未经用户明确确认，禁止任何可能导致内容丢失或不可恢复的操作，包括但不限于：删除、清空、整文件覆盖式重写、`git reset --hard`、`git checkout --`、`git clean`；尤其要警惕 `untracked` 文件上的高危操作
- 对 `issues/*`、`docs/*` 等关键决策载体，优先增量更新，避免 Delete + Add 覆盖式改写；非必要或未经用户明确指示，不得整文件重写
- 不确定操作是否会造成内容丢失，或影响范围是否明显扩大时，必须先询问用户；尽量先评估，再执行

## 六、可观测性要求（强制）
- 任何“策略、护栏、开关、限制”一旦会**无声地**改变用户体验，例如不触发推理、不派发 relay、不回复、不写入 session、不执行某个分支，都必须补齐可观测性
- 必须有结构化日志，并带 `reason_code`，让排障不依赖猜测或倒推
- 结构化日志至少应包含：`event`、`reason_code`、`decision`、`policy`
- 上下文允许时，再补：`session_key`、`channel_type`、`tool_name`、`remediation`，避免后续各写各的
- 日志级别要可控，避免过度噪声；优先仅在“命中候选但被策略拦截或降级”时记录；必要时加简单去重或限频
- 日志不得打印敏感字段，例如 token、完整正文；正文如需辅助排障，只能记录短预览或哈希
- 凡是命中 strict one-cut 或硬切换规则，并在**现有主路径**上直接拒绝的 legacy 输入、配置或持久化形状，必须留下结构化拒绝日志；禁止为了留痕而新增 legacy 专用识别分支

## 七、测试要求（强制）
- 关键 issue（无论功能还是修复），原则上必须配套测试用例，优先单元测试 / 集成测试（Unit / Integration）
- 测试必须**精简且完备**：聚焦关键功能或修复的主路径、关键边界、关键失败面；不要堆大量重复用例去证明 legacy 已不再支持、fallback / alias 已被移除等已冻结约束
- 对严格拒绝 / 策略拦截（strict reject / policy block）类改动，只保留少量能证明“拒绝生效 + `reason_code` / 强告警可观测 + 无 silent degrade”的关键用例
- 若确实无法自动化覆盖，必须在 issue 中明确记录：
  - 为什么无法自动化（缺口原因）
  - 手工验收步骤与验收口径

## 八、开工前冻结与防返工（强制）

### 8.1 未冻结，不开工
- 对 P0 / P1、one-cut、硬切换、重构收口类任务，若以下事项尚未冻结，不得直接铺开实现：
  - 唯一事实来源（owner）是谁
  - 边界在哪里，哪些层可以改，哪些层不得外溢
  - 哪些 legacy 必须直接删除，哪些输入必须直接拒绝
  - 可观测性口径是什么，拒绝 / 降级时要记录什么 `reason_code`
  - 验收标准与核心测试面是什么
- 若 issue、设计文档、实现计划之间仍存在口径冲突、职责不清、范围缺口、未决分歧，必须先补文档、补冻结项、补主单，再写代码。

### 8.2 先把尾巴找全，再做 one-cut
- 开始 one-cut 之前，必须先把所有相关旧路径、旧字段、旧概念、旧文档残留、旧测试口径尽量找全；禁止边实现边补漏、边 review 边临时盘点。
- 对明确要求“切干净”的任务，只有完成一轮面向关键词和主要消费点的全局 inventory / review，才能宣称范围已收敛。
- 若后续 review 发现还有系统性漏网点，优先回到 issue / 主单补齐范围、契约、测试计划，再继续实现；不得只在代码里局部打补丁掩盖问题。

### 8.3 主单必须是唯一准绳
- 一个实现闭环只能有一个主 issue / 主文档作为最终准绳；其它 survey、讨论稿、补缺单、独立 spec，必须明确主从关系。
- 若多份文档同时指导实现，且口径不一致，必须先收敛文档，再推进代码；禁止让实现者和 reviewer 在多份文档之间来回猜口径。
- 评审中新增的重要规则、边界、拒绝语义、测试要求，必须先并回主单或明确的唯一事实源文档，再进入下一轮实现。

### 8.4 一次改动必须追求整链闭环
- 修复或重构不能只改单层表象；必须检查是否同时涉及代码路径、配置入口、UI / RPC 契约、结构化日志、测试、文档。
- 若问题本质是契约错位、owner 分裂、边界漂移、生命周期语义错误，就必须修根因；不得只修用户最先看到的那一层表象。
- 非必要不得顺手扩大范围；但一旦某个相邻层是当前闭环必需项，就必须在本轮一起收口，不能故意留成下一轮返工。

### 8.5 先钉核心路径，再写实现
- 关键任务应优先把核心主路径、关键边界、关键失败面、关键回归点冻结为测试或明确验收口径，再进入实现。
- 若 review 已经指出阻塞性回归或系统性缺口，优先补 failing tests / 回归测试或补明确验收，不得绕过验证直接继续写功能。
- 只有当“规则已冻结、测试面已冻结、文档已对齐”同时成立时，才可以宣称该任务具备“一次性做到位”的基础。

## 九、语言风格要求

- 禁止在回复中使用中国互联网大厂黑话，输出务必专业客观、准确易懂。
- 禁止出现下列班味黑话：抓手、痛点、对齐、拉通、闭环、踩坑、拍板、一刀、狠一点、稳稳接住、收口、吃/不吃等。
