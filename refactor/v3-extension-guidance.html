V3 扩展风险与指导原则

本文档只定义一件事：

  • 在第三版设计继续推进时，哪些地方最容易重新失控，以及当前应先遵守哪些指导性原则

本文档不讨论：

  • 某一条原则现在就要不要立刻变成硬约束
  • 具体代码应该怎么改
  • 某个渠道今天的局部实现细节
  • 最终版字段表、接口表、测试表的冻结版本

也就是说，本文档讨论的是：

  • 后续扩展时，先要警惕哪些失控方向
  • 当前先按什么原则约束设计讨论

而不是：

  • 现在就把所有规则一次性钉死

一句话结论

第三版现在最大的风险，不是方向不清，而是：

  • 文档层已经收敛
  • Telegram 这条主链的代码契约已基本冻结（除落盘替换之外）
  • 但保存层替换（session_event + migration）与全渠道通用契约仍未冻结

所以当前最重要的不是继续长新概念，而是先记住几条扩展期指导原则，避免后面一边实现、一边把边界重新写散。

一句话：

  • 先把容易失控的方向标出来，先按指导原则推进，具体硬约束后面再逐条定

这份文档为什么现在就要有

OpenClaw 已经比当前代码更早进入“多渠道、多入口、多策略、多特例”的阶段。

它的现状说明了一件事：

  • 架构在纸面上清晰，不等于扩展后还能一直保持清晰

很多问题都不是设计一开始就错了，而是系统变大后慢慢长出来的，例如：

  • session_key 开始承担越来越多运行时语义
  • adapter 又重新参与“给模型的文本”塑形
  • 显式 override、特例入口、兼容逻辑越来越多
  • 平台表面对象慢慢重新渗进 core

所以第三版现在需要的，不是更多抽象概念，而是：

  • 一组指导后续讨论的防失控原则

OpenClaw 现状带来的直接启发

这里只提启发，不做价值判断。

1. session_key 很容易从“编码结果”变成“运行时语义载体”

当系统变复杂后，最容易出现的情况是：

  • 一开始只是拼 key
  • 后来很多逻辑开始反向解析 key
  • 最后代码对 key 的字符串结构产生隐式依赖

这会带来两个问题：

  • key 一旦改格式，影响面会很大
  • 原本属于语义层或 adapter 的东西，会被偷偷塞回 key 结构里

2. adapter 很容易重新长出 prompt 形态字段

系统一旦进入复杂渠道阶段，adapter 往往会觉得：

  • “我这里最懂渠道语境”
  • “我直接先把文本整理好更方便”

于是就容易重新长出：

  • 面向模型的临时文本字段
  • 某渠道专属的上下文包装
  • 事实记录与模型输入混写的中间结构

这类东西短期很顺手，长期会把边界重新打穿。

3. 平台表面对象会不断向 core 施压

群、频道、thread、topic、reply-root、forum topic，这些对象在不同渠道里都存在。

系统一开始很容易说：

  • “先抽象成一个统一概念”

但扩展到一定阶段后，真正的压力通常来自：

  • 它们并不完全等价
  • 它们又都想参与分桶、上下文、回复路由、权限策略

如果处理不好，core 就会重新被平台细节拉脏。

4. 显式 override 会越来越诱人

一旦系统支持：

  • 外部直接指定 session_key
  • hook / tool / relay 直接覆盖路由目标
  • 特定场景跳过默认分桶逻辑

短期上看很灵活，长期上看最容易把主链语义绕穿。

这类能力不是不能有，而是要非常克制。

5. 复杂群策略最容易污染基础会话语义

真正把系统拖复杂的，往往不是普通 DM，也不是普通 group。

最容易把边界冲散的，是这些东西：

  • mention
  • listen-only
  • relay
  • mirror
  • thread/topic 特例
  • reply_ref
  • 多媒体和语音

如果这些东西直接改写分桶、会话记录、上下文拼接，系统很快就会重新回到第二版状态。

当前建议先遵守的指导原则

这些原则当前只作为讨论与设计原则。

现在先不要把它们理解成全部都已冻结的强规则。

1. session_key 先坚持“只编码，不承载解释职责”

当前建议始终按这个方向讨论：

  • session_key 是编码结果
  • 它用于稳定命中同一个逻辑桶
  • 它不应成为上层语义定义来源

进一步说：

  • 能在 resolver 阶段决定的语义，不要留到后面靠拆 key 再理解
  • 能由 adapter 返回的结构化结果，不要只编码进 key 再指望下游去猜

这条不是说以后永远绝对不能解析 key，而是说：

  • 不要把“反向解析 key”当成常规主路径

2. adapter 先坚持“交事实，不交最终模型文本”

当前建议始终按这个方向讨论：

  • adapter 负责原生对象解析
  • adapter 负责结构化归一化
  • adapter 负责渠道收发协议

但 adapter 不应天然负责：

  • 最终给模型的文本
  • 长期保存格式
  • 某个渠道自己的 transcript 语义

换句话说：

  • adapter 可以提供事实
  • adapter 可以提供少量补充信息
  • 但不应主导最终上下文长什么样

3. session_event 先坚持“宁可少一点，也不要重新混层”

当前建议始终按这个方向讨论：

  • session_event 首先要稳定
  • 其次才是丰富

也就是说：

  • 不要一开始就把各种渠道细节都灌进去
  • 也不要为了省事，重新把“给模型的文本”直接塞进去

如果字段是否应进入统一事件记录层还拿不准，当前更稳妥的做法通常是：

  • 暂时留在 adapter 的补充信息里
  • 后面再讨论它是否真的值得上升为统一字段

4. core 先坚持“少量语义 + 明确主链”

当前建议始终按这个方向讨论：

  • core 只保留少量稳定概念
  • 不因为某个渠道当前实现方便，就扩张 core 总表

尤其要警惕：

  • 为了容纳某个渠道特例，往 core 里硬塞新名词
  • 为了兼容历史逻辑，长期保留混合概念对象

第三版真正要守住的不是“概念数量一定最少”，而是：

  • 主链必须明确

也就是：

  • 会话语义怎么定
  • 会话桶怎么定
  • 统一事件怎么写
  • 上下文怎么整理

这四件事不能再混。

5. 群复杂策略先坚持“附着在主链外，不改写主链定义”

当前建议始终按这个方向讨论：

  • 群复杂策略可以很多
  • 但基础会话语义主链必须先单独成立

也就是说：

  • 先明确 group 的分桶与上下文基础语义
  • 再让 mention / relay / mirror / listen-only 这类策略附着其上

而不是反过来：

  • 先由复杂策略决定会话该怎么理解

6. 显式 override 先坚持“少、窄、可观测”

当前建议始终按这个方向讨论：

  • 显式 override 可以存在
  • 但应始终被视为例外能力，而不是默认主路径

更准确地说，后面讨论这类能力时，至少要同时问三件事：

  • 它为什么不能走默认主链
  • 它的生效边界是否清楚
  • 它是否会让排障变得困难

如果这三件事讲不清，说明 override 很可能还不适合进入主链。

7. 新渠道接入时，优先验证“是否会把 core 拉脏”

第三版虽然当前先从 Telegram 开刀，但后面一定还会遇到别的渠道。

到那个阶段，最重要的检查点不是：

  • “能不能快速接上”

而是：

  • “这个渠道是否正在逼 core 接受它的内部概念”

如果答案是“是”，就说明边界设计还不够好。

当前最值得优先继续讨论的几个问题

这几个点现在都不建议直接钉死，后面应逐条讨论。

1. session_key 的允许能力边界

后面需要继续讨论：

  • 哪些场景允许显式指定 session_key
  • 哪些场景只允许 resolver 产出
  • 运行时允许在多大程度上从 key 回推信息

2. adapter 输出对象的最小稳定契约

后面需要继续讨论：

  • TG adapter 到底最少要输出哪些字段
  • 哪些字段属于统一输入
  • 哪些字段只能算补充信息

3. session_event 的最小必需字段集

后面需要继续讨论：

  • 哪些字段必须先冻结
  • 哪些字段可以晚一点再升格
  • 哪些字段永远不该进入统一事件记录层

4. 群复杂策略和基础 group 语义的边界

后面需要继续讨论：

  • mention、listen-only、relay、mirror 到底各自位于哪一层
  • 它们能不能影响分桶
  • 它们能不能影响统一事件记录
  • 它们能不能影响上下文整理

5. 出站语义结果的稳定边界

后面需要继续讨论：

  • core 的“回复结果”最少应该表达什么
  • 哪些东西交给 adapter 决定
  • 哪些东西不能再让渠道自由发挥
